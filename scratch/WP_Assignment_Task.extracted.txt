1. Problem Statement
Develop a full-stack web application for registering, assigning, tracking, resolving, and
analyzing complaints raised for a specific area, campus, department, or facility. The
application must support the complete complaint life cycle from submission to closure
and must not be limited to simple CRUD pages.
The system should include role-based login, area/category-wise complaint registration,
complaint workflow and status tracking, evidence upload, staff assignment, SLA-based
escalation, dashboard and reports, and at least one JSON API built in PHP.
2. Technology Coverage Requirements
The project must demonstrate coverage of the major topics of the syllabus through the
following components:
•Frontend using HTML5, CSS, JavaScript, and responsive design principles.
•Server-side programming using PHP.
•Database programming using MySQL with proper CRUD operations.
•Client-side and server-side validation.
•State management using sessions and at least one cookie-based feature.
•File upload handling for complaint proof and action proof.
•Asynchronous operations using AJAX and/or jQuery.
•At least one PHP-based JSON API endpoint.

Requirement Analysis and Scope Definition
•Define the scope of your complaint system according to your assigned complaint do-
main.
•Identify the users of the system and describe their responsibilities.
•Write the business rules and the expected outputs of the system.
Database Design
•Design a normalized relational database for the project.
•Suggestedtablesincludeusers,roles,complaint_categories,area_master,complaints,
complaint_attachments,complaint_history,assignments,status_master, and
feedback_or_verification.
•Use primary keys, foreign keys, and proper relationships.
•Store complaint history in a separate table.
Authentication and Authorization
•Develop login and logout functionality with at least three roles: Complainant/User,
Staff/Operator, and Supervisor/Admin.
•Use PHP sessions for authentication and authorization.
•Use cookies for one supporting feature such as saved filters, selected theme, or recently
selected area.
Complaint Category and Area Master
•Create complaint categories according to the assigned complaint domain.
•Create the area hierarchy according to the assigned area model and mapped location
structure.
•Provide add, update, disable, and list operations for categories and area records.
•Use dependent dropdowns wherever required.

Complaint Registration Module
•Develop a complaint registration form containing complaint title, description, cate-
gory, area, exact location, priority, complaint date/time, complainant information,
and image/document upload.
•Implement client-side validation using JavaScript and server-side validation using
PHP.
•Generate a complaint ID and implement duplicate complaint detection.
•Validate file type and file size.
Complaint Workflow and Status History
•Implement a complaint workflow with statuses such as Submitted, Verified, Assigned,
In Progress, Resolved, Closed, Reopened, and Escalated.
•Every status update must be recorded incomplaint_historywith timestamp, up-
dater, and remark.
•Provide a complaint timeline.
•Prevent invalid status transitions.
Assignment and SLA Handling
•Allow the Supervisor/Admin to assign complaints to staff.
•Ensure that staff members can only access complaints assigned to them.
•Implement Initial Response SLA and Resolution SLA using the student-specific for-
mulas.
•Highlight complaints that cross SLA limits and show escalated complaints separately
in the dashboard.
•If U is odd, repeated complaints from the same area and category within 7 days must
be flagged.
•If U is even, reopened complaints must require supervisor approval before reassign-
ment.
